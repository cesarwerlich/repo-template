- [ ] **Step 1: Add Project vs Global section to PLAYBOOK.md**

After the existing "## 9. Subagents / Agent Roles" section, add:

```markdown
---

## 10. Project vs Global Configuration

- **Function:** Separate what the whole team shares (project-committed) from what each developer or agent configures for themselves (global).
- **Goal:** Team members and agents follow the same project contract without being forced into identical personal workflows.
- **Why we built it:** Conflating project rules with personal preferences causes friction — either the project enforces too much (blocking personal tooling) or too little (agents behave inconsistently across team members).
- **How we built it:** Project-level config lives in committed files (`AGENTS.md`, `.claude/agents/`, `.claude/settings.json`, `.claude/commands/`). User global config lives in `~/.claude/` and is never committed. The template only ships the project-level layer.
- **What we expect:**
  - `AGENTS.md` and `.claude/agents/` define the team's shared contract — committed, reviewed, versioned.
  - `.claude/settings.json` scopes allowed tools and hooks to the project; personal tool permissions stay in `~/.claude/settings.json`.
  - Generic reusable personas (code-reviewer, planner, security-auditor) live in `~/.claude/agents/` and are available across all projects.
  - Project-specific personas (tech-lead-<projectname>, domain-expert) live in `.claude/agents/` and are committed.
  - Personal coding style, language preferences, and communication rules live in `~/.claude/CLAUDE.md` — never in the project repo.

| Artifact | Level | Why |
|---|---|---|
| `AGENTS.md` | Project | Team-shared contract |
| `.claude/agents/<project-persona>.md` | Project | Project-specific roles |
| `.claude/commands/` | Project | Project-specific slash commands |
| `.claude/settings.json` (allowed tools, hooks) | Project | Scoped to repo scripts |
| `~/.claude/CLAUDE.md` | Global | Personal style |
| `~/.claude/agents/<generic>.md` | Global | Reusable across projects |
| `~/.claude/rules/` | Global | Cross-project standards |
| `~/.claude/settings.json` (model, theme, personal perms) | Global | Personal defaults |
```

```markdown
# Agent Roster

Define the agent roles for this project. Each role gets a persona file in `.claude/agents/<role-name>.md`. This file is the index.

## How to Use

1. Fill in the roles below. Delete rows that don't apply.
2. For each role you keep, create `.claude/agents/<slug>.md` using the template at the bottom of this file.
3. Update `AGENTS.md` to reference the roles so agents know who to hand off to.

## Roster

| Role | Slug | Scope | Session start reads | Out of scope |
|---|---|---|---|---|
| Tech Lead / Orchestrator | `tech-lead` | Architecture, task breakdown, final review | CONTEXT.md, TASKS.md, DECISIONS.md | Direct implementation |
| Backend Engineer | `backend` | API, data layer, services | CONTEXT.md, TASKS.md, relevant src/ files | Frontend, infra |
| Frontend Engineer | `frontend` | UI, components, styles | CONTEXT.md, TASKS.md, relevant src/ files | API design, infra |
| Security Reviewer | `security` | Auth, secrets, input validation, CVEs | SECURITY.md, .agents/references/security-checklist.md | Feature work |
| DevOps / Infra | `devops` | CI/CD, deployment, observability | docs/operations.md, docs/release.md | Application code |

Delete unused rows. Add project-specific roles (e.g., `ml-engineer`, `data-pipeline`).

## Persona File Template

Create `.claude/agents/<slug>.md` for each role:

```markdown
# [Role Name]

## Identity
You are the [role] for [Project Name]. Your job is [one sentence].

## Session Start
Read in order:
1. `CONTEXT.md` — understand the system
2. `TASKS.md` — find your active task (filter by your role)
3. [any role-specific docs, e.g. `docs/operations.md` for devops]

## Scope
You may: [list what this role can change]
You must not: [list what is out of scope — defer to another role]

## Hand-Off
When done: update your row in `TASKS.md`, fill the Handoff block, notify [tech-lead or orchestrator].
When blocked: add blockers to your TASKS.md row, hand off with a clear "blocked on X" note.
```

## Tech Lead Responsibilities

The tech lead (or orchestrator agent) is responsible for:
- Breaking work into tasks and assigning rows in `TASKS.md`
- Reviewing output from worker agents before merging
- Resolving architectural questions by writing an ADR in `docs/adr/`
- Updating `DECISIONS.md` when a decision is accepted

# Evals

## When to Use This File

Use `docs/evals.md` when your project has **repeatable, measurable quality checks** that go beyond pass/fail unit tests:

- LLM output quality (accuracy, tone, refusal rate, latency)
- Model comparison (A vs B on a fixed dataset)
- Regression benchmarks (does this change make the model worse?)
- Human-in-the-loop scoring pipelines
- Dataset-driven fixtures stored in `evals/`

Use `docs/testing.md` for everything else: unit tests, integration tests, E2E, coverage targets, CI setup.

If you're not sure, start with `docs/testing.md`. Promote to evals only when you need reproducible benchmark datasets or score tracking over time.
